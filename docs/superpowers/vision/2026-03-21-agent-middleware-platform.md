# Dooz Agent Middleware Platform Vision

**Created:** 2026-03-21  
**Status:** Exploration / Future Direction  
**Type:** Architecture Vision

---

## Core Vision

Dooz 作为未来 Agent 的**中间件平台**，为各类智能体提供交流、工作和能力分化的基础设施。

### 定位转变

| 维度 | 当前定位 | 目标定位 |
|------|----------|----------|
| **角色** | WebSocket 消息中继服务器 | Agent 交互与调度平台 |
| **能力** | 基础消息路由、离线消息队列 | 引导子 Agent 自动分化 |
| **复杂度** | 简单调度 | 简单调度 + 分化引导 |
| **Agent 关系** | 无 | 平台与子 Agent |

---

## 设计理念

### 1. 平台角色
- Dooz **本身只做最基本的调度和交互**
- **不实现复杂 Agent 能力**
- 专注于：连接管理、消息路由、任务分发、状态同步

### 2. 子 Agent 生态
支持任意强大的智能体作为 **Sub Agent** 接入：

| 示例 Agent | 描述 |
|------------|------|
| OpenClaw | 通用智能体 |
| AstrBot | 机器人框架 |
| ... | 其他任意 Agent |

这些 Agent **不需要知道平台内部运作**，只需遵循平台的通信协议。

### 3. 自动分化逻辑
**核心目标：** 通过平台引导实现子 Agent 的**自动分化**

- **提高并行度**：任务自动分解，并行执行
- **减少信息差**：统一调度，避免重复工作

**待解决问题：** 自动分化的具体逻辑尚未设计

## 待解决问题

### 高优先级
1. **自动分化逻辑** - 如何判断何时分化？如何选择分化策略？
2. **协议标准** - 子 Agent 与平台的通信协议设计
3. **状态一致性** - 多 Agent 协作时的状态同步机制

### 中优先级
4. **能力注册** - Agent 如何声明自己的能力？
5. **任务分解** - 平台如何将任务分解给不同 Agent？
6. **结果聚合** - 多 Agent 结果如何合并？

### 低优先级（探索阶段）
7. **动态调度** - 基于负载/能力的动态任务分配
8. **容错机制** - Agent 失败时的任务迁移
